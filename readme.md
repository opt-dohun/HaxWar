## 1. gRPC와 WebSocket 혼용 구성

### 기술적 고민
해당 프로젝트에서는 두 가지 통신 패턴이 존재하였습니다. 첫 번째는 매치메이킹처럼 요청-응답 구조의 단발성 통신이고, 두 번째는 인게임 명령처럼 빈번하게 발생하는 실시간 통신입니다. 두 패턴은 요구사항이 정확성과 스키마 안정성, 그리고 지연 시간과 연결 유지, 메시지 유연성 등으로 달랐기 때문에, 하나의 프로토콜로 통일하기보다 각 패턴에 가장 적합한 프로토콜을 선택하는 것이 전체 성능과 안정성을 높일 수 있다고 판단했습니다.

### gRPC를 선택한 이유 — 매치메이킹
매치메이킹은 다음과 같은 특징이 있다고 판단했습니다.
- 대기열 등록, 매칭 결과 수신 등 정형화된 요청-응답 구조
- 클라이언트-서버 간 데이터 전송 객체 규격이 명확해야 동작 오류를 방지 가능
- 연결이 짧고 빈번하지 않으며, 정확한 응답이 중요

이에 gRPC를 채택하여 Protobuf로 스키마를 선언하고, 서버와 클라이언트가 동일한 데이터 전송 객체를 사용하도록 강제했습니다. 이를 통해 스키마 불일치로 인한 오류를 원천 차단할 수 있었고, 서버 단방향 스트리밍을 활용해 매칭 완료 이후 필요한 상태 정보를 클라이언트에 전달하는 방식으로도 확장할 수 있다고 판단하였습니다.

### WebSocket을 선택한 이유 — 게임 세션
게임 세션은 다음과 같은 특징이 있다고 판단했습니다.
- 유닛 이동, 공격, 턴 종료 등 작은 크기의 명령을 빈번하게 송수신
- 빈번한 요청 특성상 연결을 오래 유지하는 편이 유리
- 지연에 민감하며, 하나의 연결에서 다양한 형태의 비정형 메시지를 처리해야 함

인게임 세션은 이동, 공격, 스킬 등 규격이 다른 비정형 메시지 요청이 빈번하게 발생합니다. gRPC 양방향 스트리밍의 경우, Protobuf가 강력한 타입 체크를 요구하기 때문에 Any 타입 등을 통해 바이너리 배열을 감싸고 다시 식별 코드에 맞춰 역직렬화해야 하는 처리 비용이 큽니다. 반면 WebSocket은 수신된 버퍼에서 사전에 약속한 식별 코드 바이트만 읽어 명령의 타입을 확인하고, 나머지 페이로드 구간을 타입에 따라 즉각적으로 역직렬화하여 처리할 수 있어 오버헤드 없는 통신이 가능합니다.

## 2. 분산 환경 동기화 아키텍쳐 구성

```
[Client A] (Node 1 접속)         [Client B] (Node 2 접속)
[Game Server Node 1]             [Game Server Node 2]
      │                                │
      └─────────► [ Redis Store ] ◄────┘
             1. JSON 형식 상태 영속화
             2. 분산 락 동시성 제어
             3. 발행 구독 기반 이벤트 브로드캐스트
```

### 문제
2,000명 이상의 대규모 사용자 수용을 위해서는 서버 수평 확장이 필수적입니다. 그러나 초기에는 각 서버가 게임 상태를 인메모리에 직접 보유하는 상태 유지 구조였기 때문에, 사용자가 서로 다른 서버 노드에 매칭될 경우 상태가 불일치하여 게임 진행이 불가능해지는 문제가 존재하였습니다.

### 행동
- 게임 세션의 전체 상태를 Redis에 JSON 형식으로 영속화하여 인메모리 기반과 동일한 접근 속도를 확보하고, 모든 서버가 상태를 직접 보유하지 않도록 구성하여 Redis를 통해 하나의 동일한 상태에 접근하도록 구성하였습니다.
- Redis에서 상태를 읽고 비즈니스 로직을 실행할 때, 분산 락을 사용하여 동일한 게임 세션 내에 사용자의 갱신 요청이 동시에 몰리더라도 순차적인 처리를 보장하여 데이터 손상을 방지하였습니다.
- 기존 메시지 큐는 경쟁 소비자 모델로, 각 서버에 동일한 이벤트를 수신해야 하는 게임 전파 구조와는 근본적으로 맞지 않다고 판단하여, Redis의 발행 구독 채널 구독자 모델을 활용하여 모든 서버가 동일한 메시지를 동시에 수신할 수 있도록 구성하였습니다.

### 결과
이 구조를 통해 서버 1에 연결된 플레이어 A의 명령 결과가 서버 2에 연결된 플레이어 B에게도 동일한 이벤트로 지연 없이 전달되고, 분산된 환경에서도 동일한 게임 경험을 제공할 수 있었습니다.

## 3. 네트워크 단절 시 시나리오 대처

```
[클라이언트 재접속 흐름도]
Client              Server (Local Memory)         Redis
│                   │                            │
├─ 1. 재연결 요청 ──►  │                            │
│  (Last Seq: 4)    │                            │
│                   ├─ 2. 원형 큐 조회            │
│                   │  (Seq 5~7 확인)             │
◄─ 3. 유실 패킷 전송   ┤                            │
│                   │  ※ Redis 입출력 차단!        │
```

### 문제
Redis의 발행 구독 모델은 구독자에게 전송 후 메모리에서 즉각적으로 삭제되는 구조로, 모바일 환경에서 흔히 발생하는 네트워크 일시 단절 시 이벤트가 유실되는 문제가 발생합니다. 이를 복구하기 위해 재접속 시마다 Redis에서 전체 이벤트를 조회하는 입출력 작업이 추가로 발생할 수 있는 상황이었습니다.

### 행동
- 서버 로컬 메모리에 고정 크기의 원형 큐를 두어 최신 이벤트 로그를 상시 적재하도록 구현했습니다. 클라이언트가 재접속하여 마지막 수신 시퀀스를 전달하면, 서버는 외부 데이터베이스 접근 없이 자신의 로컬 원형 큐에서 누락된 이벤트만 필터링하여 즉시 동기화합니다.
- 원형 큐의 크기를 고정하고, 버퍼가 가득 찰 경우 가장 오래된 데이터를 덮어씀으로써 메모리 사용량을 일정하게 유지하였습니다. 이를 통해 장기간 이벤트 누적 시 추가 공간 확장으로 인해 발생할 수 있는 힙 메모리 파편화 현상과 이를 정리하기 위한 가비지 컬렉션의 부담을 최소화할 수 있었습니다.

### 결과
Redis 발행 구독 모델이 제공하지 않는 재전송 보장을 로컬 메모리 기반으로 보완하여, 네트워크 일시 단절로 인해 이벤트가 유실되어도 복구가 가능한 구조를 설계하였습니다.

## 4. 가비지 컬렉션 최적화 설계 작업

### 문제
2,000명이 동시에 실시간 조작을 가하는 부하 상황에서, 초당 수만 건의 수신/전송 패킷 처리로 인해 메모리 할당이 빈번하게 발생하는 문제가 발생하였습니다. 부하 테스트 결과 5분 만에 2세대 Full 가비지 컬렉션이 7회 발생하여, 런타임 환경의 자원 정리 동작으로 인해 애플리케이션 실행이 일시 중지되는 지연 현상이 확인되었습니다.

### 접근
가비지 컬렉션 오버헤드는 실시간 게임 서버의 치명적인 병목 원인이자 서버의 최대 약점이라고 생각합니다. 매 요청마다 힙 영역에서 객체를 할당하고 폐기하는 구조를 제거하고, 기존에 선언한 버퍼를 재사용하거나 메모리 주소를 직접 참조하는 설계가 필요하다고 판단하였습니다.

### 행동
- 소켓 수신부에서 매번 새로운 바이트 배열을 선언하던 구조를 `ArrayPool`로 변경하였습니다. 버퍼를 메모리 풀에서 임대하고 사용 후 반납하는 방식으로 전환하여 0세대 가비지 컬렉션 발생 빈도를 낮추었습니다.
- 송신 측에서는 생성한 패킷을 `ArraySegment<byte>`로 래핑하여 전달하는 과정 대신 `Memory<byte>` 타입으로 페이로드의 메모리 주소를 직접 참조하여 전달함으로써 중간 객체가 생성되지 않도록 구축하였습니다.
- 수신된 메시지의 역직렬화 과정에서 `MemoryStream` 같은 중간 스트림 객체를 생성하지 않고, 버퍼의 일부 영역을 `ReadOnlySpan<byte>`로 직접 전달하는 고속 처리 경로를 구축하였습니다.

### 결과
부하 테스트 비교 결과 시스템 중단을 유발하는 2세대 가비지 컬렉션 발생 횟수를 7회에서 1회로 85.7% 감소시켰으며, 평균 지연 시간을 0.03ms로 최적화하였습니다. 2,000 세션 활성화 상태에서도 실제 서버의 가비지 컬렉션 힙 크기를 97.89MB로 방어하여, 장시간 구동에도 안정적인 성능을 유지하는 것을 확인하였습니다.

## 5. Nginx 및 OS 커널 레벨 커넥션 최적화

### 문제
단일 인스턴스에서 2,000명 동시 접속(웹소켓 프록시 터널링 시 클라이언트-Nginx, Nginx-백엔드 간 총 4,000개 이상의 소켓 소모) 환경을 구성할 때, Nginx의 기본 `worker_connections` 제한(1024) 및 리눅스 프로세스당 파일 디스크립터 한도(`nofile`) 설정으로 인해 연결이 드랍되는 물리적 한계가 존재했습니다. 또한, 짧은 시간에 대량의 접속 요청(Connection Spike)이 집중될 때 OS 커널의 TCP 연결 대기열이 가득 차 네트워크 타임아웃이 발생할 우려가 있었습니다.

### 행동
- **Nginx 워커 리소스 확장:** 워커당 동시 커넥션 한도를 `10240`으로 대폭 확장하고, 프로세스 파일 디스크립터 상한선(`worker_rlimit_nofile`)을 `20480`으로 동기화했습니다. 또한, 대규모 소켓 이벤트를 효율적으로 관리하도록 비동기 이벤트 루프인 `use epoll;`을 명시했습니다.
- **백엔드 커넥션 킵얼라이브:** Nginx 업스트림 설정에 `keepalive 32;`를 도입하여 백엔드 서버들과의 유휴 TCP 커넥션을 유지함으로써 매 요청마다 3-Way Handshake를 반복하는 오버헤드를 제거했습니다.
- **Docker Compose 가상 네트워크 커널 튜닝:** Docker 컨테이너의 네트워크 네임스페이스 격리 커널 파라미터를 튜닝하여 병목을 차단했습니다.
  - `net.core.somaxconn = 65535` 및 `net.ipv4.tcp_max_syn_backlog = 8192`를 주입하여 TCP Accept/SYN 대기열 한도를 대폭 확장하고 접속 폭주를 커널 단에서 수용했습니다.
  - `net.ipv4.ip_local_port_range = 1024 65000` 및 `net.ipv4.tcp_tw_reuse = 1`을 적용해 `TIME_WAIT` 소켓을 즉시 재사용하고 아웃바운드 포트 고갈을 예방했습니다.

### 결과
2,000명의 동시접속 웹소켓 부하 상황에서도 Nginx가 단 한 건의 커넥션 드랍 없이 실시간 터널링을 안정적으로 보장했습니다. 또한, 초기 핸드셰이크가 순간적으로 집중되는 상황에서도 커널 백로그 유실 없이 균일하고 짧은 지연 시간을 보여주는 고성능 인프라 아키텍처를 검증하였습니다.
