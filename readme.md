# 백엔드 개발자 김도훈 포트폴리오

## 분산 기반 실시간 멀티플레이 게임

**GitHub:** https://github.com/dohun-kim-backend/HaxWar 

성능 목표치 **:** 1,000개의 게임 룸, 2,000명의 동시 접속 환경을 타깃으로 한 Stateless 기반 실시간 분산 서버. 

## 전체 아키텍쳐 구조도

### 1. 분산 환경 동기화 아키텍쳐 구성

```
[Client A] (Node 1 접속)         [Client B] (Node 2 접속)
      │                                │
      ▼                                ▼
[Game Server Node 1]             [Game Server Node 2]
      │                                │
      └─────────► [ Redis Store ] ◄────┘
             1. 상태 영속화 (JSON)
             2. 분산 락 (SetNX) 동시성 제어
             3. Pub/Sub 이벤트 브로드캐스트
```

**문제** 

2,000명 이상의 대규모 유저 수용을 위해서는 서버 스케일링이 필수적일 것입니다. 그러나 초기에는 각 서버가 게임 상태를 인메모리에 직접 보유하는 Stateful 구조였기 때문에, 사용자가 서로 다른 서버 노드에 매칭될 경우 상태가 불일치하여 게임 진행이 불가능해지는 문제가 존재하였습니다.

**행동**

- Game 세션의 전체 상태를 Redis에 JSON으로 영속화하여, 인메모리 기반과 동일한 접근 속도를 확보하고, 모든 서버가 상태를 직접 보유하지 않도록 구성하여, 각자 다른 서버에서 매칭되어 요청이 분산되어도 Redis를 통해 하나의 동일한 상태에 접근하도록 구성하였습니다.
- Redis에서 상태를 읽고 비즈니스 로직을 실행할 때, SetNX 분산락을 사용하여 하나의 사용자만 접근하도록 수정하여 데이터가 손상되는 것을 방지하였습니다.
- RabbitMQ 등 기존 메시지 큐는 경쟁 소비자 모델로, 각 서버의 Worker들이 하나의 큐에서 메시지를 나눠 가져가기 때문에 각 서버에 동일한 이벤트를 수신해야 하는 게임 전파 구조와는 근본적으로 맞지 않다고 판단하여, Redis pub/sub 채널의 구독자 모델을 활용하여 모든 서버가 동일한 메시지를 동시에 수신할 수 있도록 구성함으로써 별도 Worker 없이도 균일한 이벤트 전파 구조를 구축하였습니다.

**결과** 

이 구조를 통해 서버 1에 연결된 플레이어 A의 명령 결과가 서버 2에 연결된 플레이어 B에게도 동일한 이벤트로 지연 없이 전달되고, 분산된 환경에서도 동일한 게임 경험을 제공할 수 있었습니다.

### 2. 네트워크 단절 시 시나리오 대처

```
[클라이언트 재접속 흐름도]
Client         Server (Local Memory)         Redis (DB)
  │                   │                          │
  ├─ 1. 재연결 요청 ──►│                          │
  │  (Last Seq: 4)    │                          │
  │                   ├─ 2. CircularBuffer 조회   │
  │                   │  (Seq 5~7 확인)          │
  ◄─ 3. 유실 패킷 전송 ─┤                          │
  │                   │  ※ Redis I/O 원천 차단!    │
```

**문제** 

Redis Pub/Sub은 구독자에게 전송 후 메모리에서 즉각적으로 삭제되는 구조로 모바일 환경에서 흔히 발생하는 네트워크 순단 시 이벤트가 유실되는 문제가 발생합니다. 이를 복구하기 위해 재접속 시마다 Redis에서 전체 이벤트를 조회하는 I/O 작업이 추가로 발생할 수 있는 상황이었습니다.

**행동** 

서버 로컬 메모리에 고정 크기의 원형 큐를 두어 최신 이벤트 로그를 상시 적재하도록 구현했습니다. 클라이언트가 재접속하여 마지막 수신 시퀀스를 전달하면, 서버는 외부 DB 접근 없이 자신의 로컬 원형 큐에서 누락된 이벤트만 필터링하여 즉시 동기화합니다.

원형 큐의 크기를 고정하고, 버퍼가 가득 찰 경우 가장 오래된 데이터를 덮어씀으로써 메모리 사용량을 일정하게 유지하였습니다. [ 이를 통해 장기간 이벤트 누적 시 추가 공간 확장으로 인해 발생할 수 있는 힙 메모리 파편화 현상과 이를 정리하기 위한 GC의 부담을 최소화할 수 있었습니다. ]

**결과** 

Redis pub/sub이 제공하지 않는 재전송 보장을 로컬 메모리 기반으로 보완하여, 네트워크 순단으로 인해 이벤트가 유실되어도 복구가 가능한 구조를 설계하였습니다. 

### 3. GC 최적화 설계 작업

**문제** 

2,000명이 동시에 실시간 조작을 가하는 부하 상황에서, 초당 수만 건의 수신/전송 패킷 처리로 인해 메모리 할당이 빈번하게 발생하는 문제가 발생할 것이라고 판단되었고, 부하 테스트 결과 5분 만에 Gen 2 Full GC가 7회 발생하여, CLR 런타임 환경에 의해 자원 정리 목적으로 애플리케이션 실행이 멈추는 지연이 확인되었습니다.

**접근** 

GC 오버헤드는 실시간 게임 서버의 치명적인 병목 원인이자 서버의 최대 약점이라고 생각합니다. 매 요청마다 힙 영역에서 객체를 할당하고 폐기하는 구조를 제거하고, 기존에 선언한 버퍼를 재사용하거나 메모리 주소를 직접 참조하는 설계가 필요하다고 판단하였습니다.

**행동** 

- 소켓 수신부에서 `new byte[]`로 매번 버퍼를 선언하던 구조를 ArrayPool로 변경하였습니다. 버퍼를 메모리 풀에서 임대하고 사용 후 반납하도록 전환하여 Gen 0 GC 발생 빈도를 낮추었습니다.
- 송신 측에서는 생성한 패킷을 `ArraySegment<byte>`로 래핑하여 전달하는 과정 대신 `Memory<byte>` 타입으로 페이로드의 메모리 주소를 직접 참조하여 전달함으로써 힙 할당을 제거하였습니다.
- 수신된 메시지의 역직렬화 과정에서 `MemoryStream` 같은 중간 스트림 객체를 생성하지 않고, 버퍼의 일부 영역을 `ReadOnlySpan<byte>`로 직접 전달하는 Fast-Path를 구축하였습니다.

**결과** 

부하 테스트 비교 결과 시스템 중단을 유발하는 Gen 2 GC 발생 횟수를 7회에서 1회로 85.7% 감소시켰으며, 평균 지연 시간을 0.03ms로 최적화하였습니다. 2,000 세션 활성화 상태에서도 실제 서버의 GC 힙 크기를 97.89MB로 방어하여, 장시간 구동에도 안정적인 성능을 유지하는 것을 확인하였습니다.

> [!IMPORTANT]
**부하 테스트 지표와 실제 서버 지표의 분리**
 
부하 테스트 리포트 하단의 `SERVER RESOURCE` 항목은 실제로는 부하 테스트 클라이언트(`HexWar.LoadTests`) 프로세스 자체의 리소스입니다. 실제 게임 서버(`hexwar-server`)의 성능 지표는 Docker 컨테이너 내부 진단 API(`/api/diagnostics/stats`)를 통해 별도로 측정하였으며, 위 표의 최적화 이후 수치는 이 실제 측정값입니다.
