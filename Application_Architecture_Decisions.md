# 애플리케이션 아키텍처 및 인프라 설계 (Application Architecture Decisions)

본 문서는 **HexWar** 프로젝트의 애플리케이션 및 인프라 레이어에서 겪은 동시성 제어, 네트워크 최적화, 메모리 누수 방지(GC 압박 최소화)와 관련된 기술적 고민 및 아키텍처 결정을 정리합니다.

## 1. 이벤트 배치 전송을 통한 네트워크 I/O 최적화

### 고민의 배경
실시간 멀티플레이 게임 서버에서 프레임이나 미세 로직 변화마다 소켓 전송(`WebSocket.SendAsync`)을 개별적으로 호출하면, 잦은 시스템 콜 및 네트워크 패킷 생성으로 인해 서버의 CPU 사용률과 레이턴시가 폭증합니다.

### 최적화 결정: 배치성 단일 이벤트 해소 (`RoundResolved`)
- **개별 전송의 최소화:** 각 플레이어가 이동 명령을 보낼 때는 출발 사실(`UnitsDeparted`)만 전달하여 전장의 안개를 형성하는 최소한의 통신만 보냅니다.
- **배치형 마감 처리:** 한 라운드가 해소될 때 발생하는 모든 결과(이동 완료, 조우 발생, 조우 결정에 따른 후퇴, 보급로 수비대 생성 등)를 별개의 소켓 메시지로 쪼개지 않고, 하나의 완성된 **`RoundResolved` 통합 이벤트** 스냅샷으로 모아서 한 번에 JSON으로 인코딩하여 전송합니다.
- **결과:** 라운드당 웹소켓 쓰기 횟수가 기존 대비 **80% 이상 절감**(10~15회 → 1회 수준)되어 네트워크 I/O 오버헤드가 급격히 감소했습니다.

## 2. 정보 가시성(Information Visibility) 계층 필터링

### 고민의 배경
전장의 안개(Fog of War)와 비대칭 정보를 특징으로 하는 게임이므로, 어떤 이벤트를 누구에게 송신할지 완벽히 구분해야 합니다. 서버 덤프나 패킷 감청을 통해 적군의 정보가 사전 노출되는 것을 원천 차단해야 합니다.

### 구현 전략: 브로드캐스트와 프라이빗 샌드의 분리
- **`BroadcastToRoomAsync` (공개 정보):** 모든 플레이어가 동등하게 알아야 하는 이벤트(예: 라운드 시작, 게임오버, 특정 진지 점령 등)만 방에 참가한 전체 소켓으로 브로드캐스트합니다.
- **`SendToPlayerAsync` (비공개 정보):** 특정 유저의 개인 화면에만 보여야 하는 정보(예: 조우 시 내가 선택한 행동 등)는 대상 플레이어의 식별자(`PlayerSide`) 소켓으로만 다이렉트 전송합니다.
- **동적 가시성 필터링:** 도메인 이벤트가 수신될 때, `Domain.Entities.TravelingGroup.ToPublicInfo()`를 적용하여 적 진영에는 목적지 정보를 아예 누락시킨 상태로 객체를 가공한 뒤 패킷을 직렬화합니다.
